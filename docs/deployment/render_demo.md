# Render demo deployment - SwifPro BI

How to host the SwifPro BI ERP on a **free Render web service + free Render
PostgreSQL** for controlled demo / manual QA via a public link.

> **Demo / QA only.** Do not load real customer, supplier, or financial data.
> The free tier sleeps when idle and the free database is time-limited (see
> *Known free-tier limitations*).

---

## What was added for deployment

- `render.yaml` - Render Blueprint (web service + Postgres, build/start/health).
- `DATABASE_URL` support in `swifpro_bi/settings.py` (via `dj-database-url`);
  falls back to local SQLite when `DATABASE_URL` is unset.
- `/health/` endpoint (unauthenticated 200 "ok", no DB access) for Render's
  health check.
- Deployment dependencies already present in `requirements.txt`: `gunicorn`,
  `whitenoise`, `psycopg2-binary`, `dj-database-url`.

No ERP/business logic was changed.

---

## Prerequisites

- A GitHub account with this repository (Render deploys from GitHub).
- A free [Render](https://render.com) account.
- The release to deploy: tag **`v0.1.0-rc1`** (or `main`).

---

## Option A - Deploy with the Blueprint (recommended)

1. Push `render.yaml` to the branch you want to deploy (already in the repo).
2. In Render: **New + → Blueprint**, connect this repo, pick the branch.
3. Render reads `render.yaml` and provisions:
   - Web service **swifpro-bi-demo** (free)
   - PostgreSQL **swifpro-bi-db** (free), wired to the web service via
     `DATABASE_URL`.
4. Click **Apply**. The build runs:
   `pip install -r requirements.txt && python manage.py collectstatic --noinput && python manage.py migrate`
5. When live, the app is at `https://swifpro-bi-demo.onrender.com` (the exact
   subdomain is shown in the dashboard).

`DJANGO_SECRET_KEY` is auto-generated by Render (`generateValue: true`) - no
secret is committed. `DJANGO_ALLOWED_HOSTS` is `.onrender.com` and
`DJANGO_CSRF_TRUSTED_ORIGINS` is `https://*.onrender.com`, so any assigned
`*.onrender.com` host works without further edits.

## Option B - Manual dashboard setup (no Blueprint)

1. **New + → PostgreSQL** (free). Note its **Internal Database URL**.
2. **New + → Web Service**, connect the repo/branch. Set:
   - **Runtime:** Python 3
   - **Build command:**
     `pip install -r requirements.txt && python manage.py collectstatic --noinput && python manage.py migrate`
   - **Start command:** `gunicorn swifpro_bi.wsgi:application`
   - **Health check path:** `/health/`
3. Add the environment variables below.

## Environment variables

| Variable | Value (demo) | Purpose |
|---|---|---|
| `DATABASE_URL` | Render Postgres Internal URL | Postgres connection (Blueprint wires this automatically) |
| `DJANGO_DEBUG` | `0` | Production mode (DEBUG off) - **required** |
| `DJANGO_SECRET_KEY` | strong random (Render-generated) | Django secret - **required when DEBUG off** |
| `DJANGO_ALLOWED_HOSTS` | `.onrender.com` | Allowed hosts (leading dot = subdomain wildcard) |
| `DJANGO_CSRF_TRUSTED_ORIGINS` | `https://*.onrender.com` | CSRF trusted origins |
| `DJANGO_DB_SSL_REQUIRE` | `0` | `1` only if connecting to Postgres over the public internet |
| `PYTHON_VERSION` | `3.12.6` | Python runtime |
| `DJANGO_SECURE_SSL_REDIRECT` | `1` (default) | Force HTTPS (already default when DEBUG off) |
| `EMAIL_HOST`, `EMAIL_HOST_USER`, `EMAIL_HOST_PASSWORD` | *(optional)* | SMTP; if unset, email features are inert in prod (console backend is dev-only) |

Generate a strong secret locally if setting it by hand:
`python -c "import secrets; print(secrets.token_urlsafe(50))"`

## Build / start / migration / static commands

- **Build:** `pip install -r requirements.txt && python manage.py collectstatic --noinput && python manage.py migrate`
- **Start:** `gunicorn swifpro_bi.wsgi:application`
- **Migrations** run as part of the build (`python manage.py migrate`). To run
  ad hoc, use the Render **Shell** for the service: `python manage.py migrate`.
- **Static files** are collected at build into `staticfiles/` and served by
  WhiteNoise (`CompressedManifestStaticFilesStorage`) - no separate CDN needed.

## Database setup

- The Blueprint creates the free Postgres instance and injects `DATABASE_URL`.
- The app auto-switches to Postgres whenever `DATABASE_URL` is present; with no
  `DATABASE_URL` it uses local SQLite (dev).
- Pre-migration check (run once before relying on the journal idempotency
  constraint - expect **zero** rows; migration `0093` also self-aborts safely if
  duplicates exist):
  ```sql
  SELECT tenant_id, ref_type, ref_id, COUNT(*) FROM core_journalentry
  WHERE ref_type IS NOT NULL AND ref_type <> '' AND ref_id IS NOT NULL AND ref_id <> ''
    AND ref_type <> 'TRANSFER_RECEIPT'
  GROUP BY tenant_id, ref_type, ref_id HAVING COUNT(*) > 1;
  ```
  (On a fresh demo DB this is trivially empty.)

## Create admin users (no Shell needed)

The Render **free tier has no Shell**, so `createsuperuser` can't be run
interactively. Instead, admin accounts are bootstrapped from environment
variables by a management command that runs automatically at the end of every
build:

```bash
python manage.py bootstrap_demo_admins
```

It is **idempotent**, **never prints passwords**, **never resets an existing
user's password**, exits cleanly when its env vars are absent, and touches no
other tenant. It creates:

- **1 owner** - Django superuser + staff + app **ADMIN** role.
- **Up to 4 demo admins** - staff + app **ADMIN** role, **not** Django
  superusers (full app access without superuser).

All accounts join a single demo tenant **"SwifPro BI Demo Ltd"** with an ADMIN
membership, which grants full, unrestricted app access. App login is at
`/login/`; the Django admin (owner only) is at `/admin/`.

### Required env vars - main owner
| Variable | Notes |
|---|---|
| `DJANGO_SUPERUSER_PASSWORD` | **Required** to create the owner; owner is skipped if unset. **Set this in the Render dashboard** (never committed). |
| `DJANGO_SUPERUSER_USERNAME` | Default `santhosh` |
| `DJANGO_SUPERUSER_EMAIL` | Default `santhoshkumarnanubala@gmail.com` |

### Optional env vars - four demo admins
| Variable | Notes |
|---|---|
| `DEMO_ADMINS_ENABLED` | `1` to create the four demo admins; anything else = owner only |
| `DEMO_ADMIN_DEFAULT_PASSWORD` | Shared password for the four; if unset, the four are skipped with a warning. **Set in the dashboard.** |
| `DEMO_ADMIN_1_EMAIL` … `DEMO_ADMIN_4_EMAIL` | A missing email skips just that one user (usernames are `admin1`…`admin4`) |

### Enable / disable the demo admins
- **Enable:** set `DEMO_ADMINS_ENABLED=1` **and** `DEMO_ADMIN_DEFAULT_PASSWORD`, then redeploy.
- **Disable creation:** set `DEMO_ADMINS_ENABLED=0` (or remove the password). Existing demo users are **not** auto-deleted - remove them manually (below).

### Rotate passwords
The command never resets an existing user's password. To rotate:
1. Update `DJANGO_SUPERUSER_PASSWORD` / `DEMO_ADMIN_DEFAULT_PASSWORD` in the dashboard.
2. Delete the affected user(s) in the Django admin (`/admin/auth/user/`).
3. Redeploy - the command recreates them with the new password.

### Delete / disable the demo admins later (do this before real production use)
In the Django admin (`/admin/`, owner login), open **Users** and either:
- untick **Active** to disable login, or
- delete the `admin1`…`admin4` users (and the **"SwifPro BI Demo Ltd"** tenant if
  it's no longer needed).
Then set `DEMO_ADMINS_ENABLED=0` so they aren't re-created on the next deploy.

> Security: these are **demo/admin** accounts. There are no hardcoded passwords
> or secrets in the repo - passwords come only from env vars. **Disable or delete
> them before using this deployment for anything real**, and never reuse
> production credentials.

## Seed clean demo data (optional)

The repo ships demo seeders (they create **sample** companies/products - fine for
a demo, but not "clean books"):

```bash
python manage.py seed_demo          # generic demo tenant + sample data
python manage.py seed_uk_demo       # UK multi-site demo
python manage.py seed_uom_demo      # UOM/conversions sample
```

For a **clean** demo instead, skip the seeders: create a superuser, then sign up
/ create a tenant through the app and enter a little data by hand. After seeding,
verify accounting health in the Shell:

```bash
python manage.py check_inventory_gl       # inventory subledger vs GL
python manage.py audit_serial_readiness   # legacy serial data check
```

(Seeded demo data may show an inventory↔GL variance because sample stock is
loaded without GL postings; the trial balance still balances. Real use should
start from clean books.)

## Known free-tier limitations

- **Web service sleeps after ~15 min idle**; the first request after sleep takes
  ~30–60s to wake. Fine for demo, not for latency-sensitive QA.
- **Free Postgres expires** after Render's free-tier window (currently ~30 days)
  and has limited storage/connections. Treat the data as disposable.
- **Ephemeral disk** - uploaded media (`MEDIA_ROOT`) is **not persistent** across
  deploys/restarts on the free tier. Don't rely on file uploads surviving.
- One free web service is single-instance; no horizontal scaling.
- Outbound email needs real SMTP env vars; otherwise email features are inert.

## Redeploy after a new tag

- **Auto-deploy:** if the service tracks `main`, pushing/merging to `main`
  triggers a build automatically.
- **Deploy a specific tag/commit:** in the Render dashboard → service →
  **Manual Deploy → Deploy a specific commit**, and pick the commit the tag
  points to (e.g. `v0.1.0-rc1` → `5fbce95`). Or point the service's branch at the
  tag's commit.
- Each deploy re-runs the build (collectstatic + migrate), so new migrations are
  applied automatically.

## Rollback to a previous tag

- **Fastest:** Render dashboard → service → **Deploys** → find a previous
  successful deploy → **Rollback to this deploy**.
- **By commit:** **Manual Deploy → Deploy a specific commit** and choose the
  prior tag's commit.
- **Migration note:** migrations `0081–0094` are **additive** (new fields,
  indexes, constraints), so rolling the *code* back is safe **without**
  un-migrating - older code ignores the newer columns/constraint. Avoid reversing
  migrations on a shared demo DB; if you must, do it deliberately in the Shell.
